## 1. Cambio de negocio — Registro y Login (auth)

### Qué cambió
Registro pasó de 3 flujos (email / RUT / username) a un **único flujo** con formulario completo. Login sigue siendo multi-identificador.

### Archivos modificados

| Archivo | Resumen |
|---|---|
| `src/users/entities/user.entity.ts` | Reemplaza `username` (obligatorio) + `name` + `identifierType` → `firstName`, `lastName`, `email` (NOT NULL), `rut` (nullable unique), `username` (nullable unique — handle), `acceptedPrivacyAt` |
| `src/auth/dto/register.dto.ts` | Nuevos campos: `firstName`, `lastName`, `rut`, `email`, `password`, `confirmPassword`, `acceptTerms`, `acceptPrivacy` |
| `src/auth/dto/login.dto.ts` | Campo `username` → `identifier` (acepta email, RUT o username) |
| `src/auth/interfaces/jwt-payload.interface.ts` | `username` → `email` en el payload |
| `src/auth/strategies/jwt.strategy.ts` | Valida `sub` + `email` en lugar de `sub` + `username` |
| `src/auth/auth.service.ts` | `register()` siempre Flow A (email verification); `login()` usa `findByIdentifierWithPassword()`; `issueTokens()` usa `user.email` |
| `src/users/users.service.ts` | Elimina `detectIdentifierType()`; `create()` acepta nuevos campos; nuevo método `findByIdentifierWithPassword()` (detecta email / RUT / username); `toPublic()` retorna `firstName`, `lastName`, `rut` |
| `src/users/dto/update-profile.dto.ts` | Reemplaza `name` → `firstName`, `lastName`; añade `username` (handle) |
| `src/auth/dto/reset-password.dto.ts` | Regex contraseña actualizado: añade requisito de carácter especial |
| `src/users/dto/change-password.dto.ts` | Idem |
| `src/auth/auth.controller.ts` | Actualiza descripción Swagger de email-verification/request |

### Cambios en DB (`schema.sql` — tabla `users`)

```
- Eliminado:  name, identifier_type
- Añadido:    first_name, last_name, rut (nullable unique), accepted_privacy_at
- Modificado: email → NOT NULL UNIQUE; username → nullable UNIQUE (handle opcional)
- Añadido:    tablas faltantes: email_verification_tokens, onboarding_state, biometric_preferences
```

### Regla de login resultante
```
identifier contiene '@'         → busca por email
identifier formato 12345678-9   → busca por rut
otro                            → busca por username (handle)
```

## 3. Gaps críticos aplicados — modelo Transaction

### Por qué
El equipo Python (externo) necesita estos campos antes de empezar a enviar movimientos desde cartolas.

### Archivos modificados

| Archivo | Resumen |
|---|---|
| `src/cashflow/enums/categorization-status.enum.ts` | **Nuevo.** Enum: `categorized` \| `pending_review` \| `uncategorized` |
| `src/cashflow/entities/transaction.entity.ts` | +3 campos: `bankDescription`, `destinationFundingSourceId`, `categorizationStatus` |
| `src/cashflow/dto/create-transaction.dto.ts` | +3 campos opcionales con validación y Swagger |
| `src/cashflow/dto/update-transaction.dto.ts` | +`destinationFundingSourceId`, +`categorizationStatus`. `bankDescription` excluido (inmutable) |
| `src/cashflow/services/transactions.service.ts` | Valida funding source destino; mapea nuevos campos en `create()` y `update()`; incluye `destinationFundingSource` en relaciones |

### Cambios en DB (`schema.sql` — tabla `transactions`)

```
- Añadido:  bank_description TEXT NULL
- Añadido:  destination_funding_source_id UUID NULL FK → funding_sources
- Añadido:  categorization_status VARCHAR(20) NOT NULL DEFAULT 'categorized'
- Añadido:  idx_transactions_categorization_status (parcial, solo != 'categorized')
- Añadido:  idx_transactions_destination_funding_source_id (parcial, solo NOT NULL)
```

### Para qué sirve cada campo

| Campo | Uso |
|---|---|
| `bankDescription` | Glosa cruda del banco — inmutable — insumo de auto-categorización y detección de recurrencia |
| `destinationFundingSourceId` | Permite modelar "viaje del dinero" (Origen → Destino). Excluye traspasos de métricas de gasto real cuando ambos IDs son del mismo usuario |
| `categorizationStatus` | Diferencia movimientos categorizados de los pendientes de revisión (baja confianza al importar cartola) |
